Promedio general

🤖 Análisis IA — Perspectiva consultor

buk.cl es el referente del grupo con la mejor base técnica: TTFB de 85ms, estructura semántica completa, y el mayor coverage de alt texts (97%). Sin embargo, la presencia de 304 imágenes en el home y un peso transferido de 1.6MB denotan un sitio sobrecargado visualmente. El error de JavaScript en template_header-2025.js (null.addEventListener) es un bug activo que puede afectar la funcionalidad del header para algunos usuarios.

Principal freno de performance: Exceso de imágenes (304) y ausencia de Organization/WebSite schema reducen visibilidad en rich results y sobrecargan el thread principal.
Principal oportunidad: Implementar schema Organization + WebSite permitiría activar sitelinks y searchbox en Google, maximizando el espacio en SERP para queries de marca.

⚡ Quick Wins — Acciones priorizadas
ALTA
Corregir TypeError en template_header-2025.js: El null.addEventListener rompe funcionalidad del header. Verificar que el elemento DOM exista antes del eventListener o mover el script al final del body.
MEDIA
Implementar schema Organization + WebSite: Añadir JSON-LD con @type Organization (nombre, logo, redes sociales) y WebSite (SearchAction para sitelinks searchbox). Impacto: mejora rich results en branded queries.

Agregar <meta name="robots" content="index, follow">: Buena práctica para declarar explícitamente la intención de indexación. Riesgo bajo, esfuerzo mínimo.

buk.pe tiene el mayor volumen de errores de consola del grupo (13) — casi todos provenientes de un servidor GTM personalizado (gtms-pe.buktechnology.com) que tiene CORS mal configurado. Esto indica que Perú usa una infraestructura de tracking diferente al resto del grupo, probablemente un servidor de colección de datos propio que no está configurado correctamente. También presenta los peores tiempos de carga: DOMContentLoaded en 1.9s y loadComplete en 4.1s.

Principal freno: La infraestructura de GTM personalizada (gtms-pe.buktechnology.com) está generando 10+ errores CORS por page load, consumiendo recursos del browser y bloqueando threads que deberían procesar el contenido.
Principal oportunidad: Resolver el CORS y optimizar el critical rendering path podría reducir DOMContentLoaded de 1.9s a menos de 1s.
